# vLLM 连续批处理 (Continuous Batching) 核心笔记

## 核心思想

** vLLM 不以“请求”为单位进行批处理，而是以“解码的单个步骤（Iteration）”为单位进行迭代式调度。它让 GPU 流水线永不停歇，动态地处理流入和流出的请求。相比静态批处理，吞吐量提升 2-4 倍，并显著降低延迟。

---

## 1. 问题根源：静态批处理的“木桶效应”

传统的静态批处理（Static Batching）效率低下。

-   **凑批次等待**：必须凑够一个批次（或超时）才发车。
-   **Padding 浪费**：对短请求进行大量填充（Padding），对齐最长请求。
-   **同步等待**：只要有一个请求没生成完，其他已完成的请求也必须等待，占用的 GPU 资源。

---

## 2. 实现原理

vLLM 的连续批处理实现依赖两大核心技术。

### 技术一：PagedAttention (物理基础)

它解决了 LLM 推理中最棘手的 **KV Cache 内存管理**问题。

-   **类比**：操作系统的**虚拟内存**和**分页技术**。
-   **工作方式**：
    1.  **内存分块 (Blocking)**：将 GPU 显存预先划分为大量固定大小的物理块 (Block)。
    2.  **逻辑与物理分离**：一个请求的 KV Cache 在逻辑上是连续的，但在物理上存储在离散的、不相邻的块中。
    3.  **块表 (Block Table)**：为每个请求维护一个“页表”，记录从逻辑 Token 到物理块的映射关系。
-   **带来的革命性改变**：
    -   **消除内存碎片**：内存利用率接近 100%。
    -   **按需分配**：无需为请求预留最大长度的内存，用多少分多少。
    -   **低成本操作**：添加或删除一个请求，只需修改块表和回收/分配块，操作成本极低，是实现动态调度的前提。

### 技术二：智能调度器 (逻辑核心)

调度器是决策大脑，在**每一个推理步骤**执行一个快速的调度循环。

-   **调度循环 (Per-Iteration Loop)**：
    1.  **释放 (Free)**：检查当前批次，立即将已完成（生成 `[EOS]`）的请求移除，并将其占用的所有物理块归还给内存池。
    2.  **分配 (Allocate)**：为批次中仍在运行的请求分配新的物理块，用于存储下一个生成的 KV Cache。
    3.  **接纳 (Admit)**：检查等待队列和空闲内存块。如果资源充足，立刻将等待中的新请求加入到运行批次中，参与到下一个推理步骤。
    4.  **抢占 (Preempt)** (高级)：当高优先级请求到来但资源不足时，可将低优先级请求的 KV Cache 从 GPU 换出到 CPU，释放资源，稍后再换入。

---

## 3. 对比总结

| 特性 | 静态批处理 (Static Batching) | vLLM 连续批处理 (Continuous Batching) |
| :--- | :--- | :--- |
| **批处理粒度** | 以**请求 (Request)**为单位 | 以**迭代步骤 (Iteration)**为单位 |
| **GPU 利用率** | **低** (因 Padding 和等待而空闲) | **高** (几乎总是在进行有效计算) |
| **内存效率** | **低** (预分配和内存碎片) | **极高** (PagedAttention 按需分配) |
| **吞吐量** | 低 | **非常高 (2-4x)** |
| **核心思想** | 木桶效应，同步执行 | 流水线作业，异步、独立完成 |
| **比喻** | 公交车 🚌 | 网约车拼车 🚗 |

**结论：** **PagedAttention (物理层)** + **智能调度器 (逻辑层)** = **高效的连续批处理**。
