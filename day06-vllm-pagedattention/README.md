# vLLM PagedAttention

## 1. 为什么需要？(传统 KV Cache 的痛点)

传统 KV Cache 管理方式存在两大致命问题，导致显存利用率极低。

*   **内部碎片:** 为每个请求预分配其**最大可能长度**的**连续**内存。即使请求很短，也占用巨大空间，导致 60%-80% 的显存被闲置浪费。

*   **外部碎片:** 请求结束后，显存中留下大量不连续的小块空闲内存。即使总剩余量很大，新请求也可能因找不到一块**足够大的连续空间**而被阻塞，降低了系统吞吐量。

## 2. 是什么？(借鉴操作系统的“分页”思想)

PagedAttention 的核心思想是借鉴操作系统管理虚拟内存的方式，将 KV Cache 的**逻辑视图**和**物理存储**解耦。

它将显存预先划分为许多**固定大小的物理块 (Physical Blocks)**，然后通过为每个序列维护一个**块表 (Block Table)**，将序列的**逻辑块 (Logical Blocks)** 映射到这些不连续的物理块上。

### 与操作系统类比

| 概念 | 操作系统 (Operating System) | vLLM PagedAttention |
| :--- | :--- | :--- |
| **逻辑单元** | 虚拟页面 (Virtual Page) | 逻辑块 (Logical Block) |
| **物理单元** | 物理页帧 (Physical Frame) | 物理块 (Physical Block) |
| **映射工具** | 页表 (Page Table) | 块表 (Block Table) |

## 3. 如何工作？(按需分配与高效共享)

PagedAttention 通过一个中央的块管理器 (Block Manager) 来实现动态、高效的内存管理。

1.  **按需分配:** 不再预分配，而是在生成每个 Token 时，按需从物理块池中申请一个块。用多少，给多少。
2.  **通过块表寻址:** GPU 计算 Attention 时，通过读取序列的块表，直接从各个不连续的物理块中抓取数据。
3.  **高效共享:** 多个序列可以共享指向相同物理块的块表条目，实现零成本的内存共享。
4.  **轻松释放:** 请求结束，其占用的所有物理块被归还到中央池中，可立即被任何其他请求复用。

### 核心优势对比

| 特性 | 传统 KV Cache | vLLM PagedAttention |
| :--- | :--- | :--- |
| **内存分配** | 一次性预分配连续大块 | 按需分配不连续的小块 |
| **内存浪费** | **严重** (内部碎片) | **几乎为零** |
| **内存碎片** | **严重** (外部碎片) | **完全解决** |
| **内存共享** | 困难 (需完整复制) | **高效** (共享块表指针) |
| **系统吞吐量** | 低 | **极高 (2-4x 提升)** |

### 核心特性：高效内存共享

这是 PagedAttention 最强大的特性，尤其适用于并行采样 (Parallel Sampling) 和束搜索 (Beam Search)。

*   **场景**: 一个 Prompt 需要生成 3 个不同的续写。
*   **实现**: 3 个序列的块表可以共享指向 Prompt KV Cache 的物理块。只有在生成各自独有的新 Token 时，才需要申请新的、独立的物理块。

```
Prompt KV Cache 存储在 [物理块#7, 物理块#18]

- 序列 A 块表: [7, 18, 101]  <-- 101 是为 A 新分配的块
- 序列 B 块表: [7, 18, 205]  <-- 205 是为 B 新分配的块
- 序列 C 块表: [7, 18, 301]  <-- 301 是为 C 新分配的块
```

这种机制极大地降低了复杂解码策略的内存开销，是 vLLM 实现超高吞吐量的关键。
