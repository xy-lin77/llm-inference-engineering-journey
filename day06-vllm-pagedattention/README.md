# vLLM PagedAttention

## 1. 为什么需要？(传统 KV Cache 的痛点)

传统 KV Cache 管理方式存在两大致命问题，导致显存利用率极低。

*   **内部碎片:** 为每个请求预分配其**最大可能长度**的**连续**内存。即使请求很短，也占用巨大空间，导致 60%-80% 的显存被闲置浪费。

*   **外部碎片:** 请求结束后，显存中留下大量不连续的小块空闲内存。即使总剩余量很大，新请求也可能因找不到一块**足够大的连续空间**而被阻塞，降低了系统吞吐量。

## 2. 是什么？(借鉴操作系统的“分页”思想)

PagedAttention 的核心思想是借鉴操作系统管理虚拟内存的方式，将 KV Cache 的**逻辑视图**和**物理存储**解耦。

它将显存预先划分为许多**固定大小的物理块 (Physical Blocks)**，然后通过为每个序列维护一个**块表 (Block Table)**，将序列的**逻辑块 (Logical Blocks)** 映射到这些不连续的物理块上。

### 与操作系统类比

| 概念 | 操作系统 (Operating System) | vLLM PagedAttention |
| :--- | :--- | :--- |
| **逻辑单元** | 虚拟页面 (Virtual Page) | 逻辑块 (Logical Block) |
| **物理单元** | 物理页帧 (Physical Frame) | 物理块 (Physical Block) |
| **映射工具** | 页表 (Page Table) | 块表 (Block Table) |

## 3. 如何工作？(按需分配与高效共享)

PagedAttention 通过一个中央的块管理器 (Block Manager) 来实现动态、高效的内存管理。

1.  **按需分配:** 不再预分配，而是在生成每个 Token 时，按需从物理块池中申请一个块。
2.  **通过块表寻址:** GPU 计算 Attention 时，通过读取序列的块表，直接从各个不连续的物理块中抓取数据。GPU 硬件非常擅长这种“Gather”（收集）操作，性能很高。
3.  **高效共享:** 多个序列可以共享指向相同物理块的块表条目，实现零成本的内存共享。
4.  **轻松释放:** 请求结束，其占用的所有物理块被归还到中央池中，可立即被任何其他请求复用。

### 核心优势对比

| 特性 | 传统 KV Cache | vLLM PagedAttention |
| :--- | :--- | :--- |
| **内存分配** | 一次性预分配连续大块 | 按需分配不连续的小块 |
| **内存浪费** | **严重** (内部碎片) | **几乎为零** |
| **内存碎片** | **严重** (外部碎片) | **完全解决** |
| **内存共享** | 困难 (需完整复制) | **高效** (共享块表指针) |
| **系统吞吐量** | 低 | **极高 (2-4x 提升)** |

### 关键特性：高效内存共享

这是 PagedAttention 最强大的特性，尤其适用于并行采样 (Parallel Sampling) 和束搜索 (Beam Search)。

*   **场景**: 一个 Prompt 需要生成 3 个不同的续写。
*   **实现**: 3 个序列的块表可以共享指向 Prompt KV Cache 的物理块。只有在生成各自独有的新 Token 时，才需要申请新的、独立的物理块。

```
Prompt KV Cache 存储在 [物理块#7, 物理块#18]

- 序列 A 块表: [7, 18, 101]  <-- 101 是为 A 新分配的块
- 序列 B 块表: [7, 18, 205]  <-- 205 是为 B 新分配的块
- 序列 C 块表: [7, 18, 301]  <-- 301 是为 C 新分配的块
```

这种机制极大地降低了复杂解码策略的内存开销，是 vLLM 实现超高吞吐量的关键。

# Tensor Parallelism (`tensor_parallel_size`) 

## 1. 是什么？

`tensor_parallel_size` 是一种模型并行技术，用于解决**单个 GPU 显存不足以容纳整个大模型**的问题。

-   **核心目的：** 让超大模型（如 70B+）能够运行起来。
-   **工作原理：** 将模型内部的巨大权重矩阵（Tensor）**水平切分**成 `N` 份（`N` = `tensor_parallel_size`），分别加载到 `N` 张 GPU 上。

---

## 2. GPU 原理

张量并行的性能严重依赖 GPU 间的通信硬件。

#### 工作流程：

1.  **切分 (Split)：** 模型权重被切分并加载到 `N` 张 GPU。
2.  **并行计算 (Compute)：** 输入数据被广播到所有 GPU，每张 GPU 在自己的权重分片上独立计算，得到一个中间结果。
3.  **通信与同步 (Communicate & Sync)：** 这是**性能瓶颈**所在。所有 GPU 必须通过高速互联交换彼此的中间结果，并执行集合通信操作（如 `All-Reduce`）来合并成最终的完整输出。

#### 关键硬件：NVLink vs. PCIe

GPU 间的通信路径决定了性能上限。

| 特性 | NVLink | PCIe |
| :--- | :--- | :--- |
| **双向带宽** | **~900 GB/s** (A100/H800) | ~64 GB/s (PCIe 5.0) |
| **性能差距** | **~14 倍** | 基准 |
| **通信路径** | **GPU ↔ GPU 直连** | GPU ↔ CPU ↔ GPU (绕远路) |
| **意义** | **使张量并行高效可行** | **使张量并行性能急剧下降** |

同一个节点内的多张 GPU 通过 NVLink 高速直连。分布在不同服务器节点上的 GPU 通过一个独立的、专用的 **高性能网络 (High-Performance Network)** 进行通信。目前业界的主流标准是 **InfiniBand (IB)**，其核心技术是 **RDMA (远程直接内存访问)**。

---

## 3. 性能与加速比 (Performance)

#### 核心结论：

-   **首要目标是“能跑”**，其次才是“跑得快”。
-   **加速比是次线性 (Sub-linear) 的**，即 `N` 张卡的性能 < `N` * 单卡性能。

#### 性能权衡：

1.  **延迟 (Latency) - 变高：**
    -   对于**单个请求**，由于引入了通信开销，完成时间会比（理论上存在的）超大单卡更长。

2.  **吞吐量 (Throughput) - 变高：**
    -   **主要优势所在**。`N` 张卡意味着总算力、总显存、总 KV Cache 容量都增加了 `N` 倍。
    -   这允许 vLLM 的连续批处理调度一个**更大的并发批次**，从而显著提升整个系统的总吞吐量（每秒处理的 tokens）。

#### 加速比递减：

-   **原因：** 通信开销是主要瓶颈。随着 `tensor_parallel_size` 增加，通信的复杂度和耗时也增加，这部分开销会逐渐抵消掉增加算力带来的好处。
-   **典型表现：**
    -   1 → 2 卡，吞吐量提升可能达到 **1.7x**。
    -   2 → 4 卡，吞吐量相比 2 卡可能只提升 **1.6x**。
    -   收益会不断递减。
